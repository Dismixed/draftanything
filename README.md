# Stim Labs

Party games for any crowd. Stim Labs builds Stim Games — starting with **Draft Anything**, a live multiplayer draft room where friends create private rooms, build a pool of items, draft in rounds, and get AI-powered judging with commentary.

---

## Prerequisites

- Node.js 20+
- pnpm 10+
- Supabase CLI (`brew install supabase/tap/supabase`)
- A Supabase project (local or hosted)
- An OpenAI API key (optional in fixture mode)

---

## Environment Variables

Copy `.env.example` to `.env.local`:

```
NEXT_PUBLIC_SUPABASE_URL=http://127.0.0.1:54321
NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY=<anon-key>
SUPABASE_SERVICE_ROLE_KEY=<service-role-key>
OPENAI_API_KEY=<openai-key>
OPENAI_MODEL=gpt-5.5
GUEST_TOKEN_PEPPER=<32-char-min-random-string>
APP_URL=http://localhost:3000
CRON_SECRET=<random-secret-for-vercel-cron>
```

- `NEXT_PUBLIC_SUPABASE_URL` / `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY` — from Supabase project settings (API section).
- `SUPABASE_SERVICE_ROLE_KEY` — from Supabase project settings (Service role key). Never expose to the client.
- `OPENAI_API_KEY` — Optional if using fixture mode. Required for AI commentary, judging, and pool generation.
- `GUEST_TOKEN_PEPPER` — At least 32 random characters used to sign guest tokens.
- `CRON_SECRET` — Used by Vercel Cron to authenticate the `/api/cron/auto-pick` endpoint. Omit for local dev.

## Local Supabase

### Start

```bash
pnpm supabase start
```

This boots local Postgres, GoTrue, Realtime, and storage containers. The first run applies all migrations automatically.

### Reset

```bash
pnpm supabase db reset
```

Drops all data and re-applies migrations from `supabase/migrations/`. Use this frequently during development.

### View local dashboard

```bash
open http://127.0.0.1:54323
```

## Generated Database Types

After modifying the schema (adding a migration), regenerate TypeScript types:

```bash
pnpm db:types
```

This writes to `lib/supabase/database.types.ts`. Do not edit this file by hand.

## OpenAI Fixture Mode vs Live Mode

The app supports two modes for AI features:

- **Live mode** (default): Set `OPENAI_API_KEY` in `.env.local`. Calls OpenAI for commentary, pool generation, and judging.
- **Fixture mode**: When `OPENAI_API_KEY` is not set (or the API returns errors), the app falls back to deterministic fixtures. Pool generation returns hardcoded mock items. Commentary returns canned responses. Judging uses weighted random selection.

Fixture mode is useful for local development and E2E tests. Most tests intentionally omit the API key.

## Deployment (Vercel + Supabase)

### Supabase (production)

1. Create a new project at [supabase.com](https://supabase.com).
2. Run migrations against it:
   ```bash
   supabase db push --linked
   ```
3. Note the project URL, anon key, and service-role key.

### Vercel

1. Install Vercel CLI: `npm i -g vercel`
2. Link the project:
   ```bash
   vercel link
   ```
3. Set environment variables:
   ```bash
   vercel env add NEXT_PUBLIC_SUPABASE_URL
   vercel env add NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY
   vercel env add SUPABASE_SERVICE_ROLE_KEY
   vercel env add OPENAI_API_KEY
   vercel env add OPENAI_MODEL
   vercel env add GUEST_TOKEN_PEPPER
   vercel env add APP_URL
   vercel env add CRON_SECRET
   ```
4. Deploy:
   ```bash
   vercel --prod
   ```

## Migration Order

Migrations run sequentially by filename prefix. The current order is:

1. `202606150001_initial_schema.sql` — tables, enums, constraints
2. `202606150002_policies.sql` — Row-Level Security policies
3. `202606150003_realtime.sql` — Realtime publication config
4. `202606150006_join_rpc.sql` — join-draft RPC
5. `202606150007_create_draft_rpc.sql` — create-draft RPC
6. `202606150008_start_pool_review_rpc.sql` — start-pool-review RPC
7. `202606150009_lock_pool_rpc.sql` — lock-pool RPC
8. `202606150010_draft_rpcs.sql` — pick, auto-pick, start-draft RPCs
9. `202606150011_judging_rpcs.sql` — defense, vote, phase-advance RPCs

To add a new migration, create a file in `supabase/migrations/` with a timestamp prefix greater than the last migration and run `supabase db reset` locally.

## Cron Configuration

The Vercel Cron Job (`vercel.json`) invokes `/api/cron/auto-pick` every minute. This endpoint:

1. Authenticates via `Authorization: Bearer <CRON_SECRET>` header.
2. Queries drafts in `DRAFTING` phase with an expired `turn_deadline`.
3. For each due draft, finds the current player and calls the `auto_pick` RPC.
4. Processes up to 10 drafts per invocation to stay within Vercel's timeout.
5. Returns `{ processed: N, errors?: [...] }`.

To test locally:

```bash
curl -H "Authorization: Bearer $CRON_SECRET" http://localhost:3000/api/cron/auto-pick
```

## Rollback Procedure

### Database

1. Identify the migration to roll back:
   ```sql
   SELECT * FROM supabase_migrations.schema_migrations ORDER BY version DESC;
   ```
2. Create a new migration that reverses the changes (never edit committed migrations).
3. Apply:
   ```bash
   supabase db push
   ```

### Application

1. Revert the application code to the previous commit:
   ```bash
   git revert HEAD
   ```
2. Deploy the reverted code to Vercel:
   ```bash
   vercel --prod
   ```

### Full rollback of a release

1. Revert the database to the state before the release migrations:
   ```bash
   supabase db reset --version <last-known-good-version>
   ```
2. Revert the app code:
   ```bash
   git revert HEAD~1..HEAD
   git push
   ```
3. Verify the app starts and Supabase tests pass:
   ```bash
   pnpm build && supabase test db
   ```
4. Deploy to Vercel:
   ```bash
   vercel --prod
   ```
